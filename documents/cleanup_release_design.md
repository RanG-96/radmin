# 收尾清理与提交设计

## 1. 背景

当前这批改动已经基本完成了“选项配置、本地优先开发、用户管理易用性改造”等目标，但仍残留少量会影响交付质量的问题：

- 运行时文案仍夹杂旧的“字典类型 / 字典项”表述。
- 后端通知创建模型里有未使用字段，`cargo build` 会持续给出警告。
- `TODO.md` 还保留上一阶段的旧术语，和当前页面命名不完全一致。

这些问题都不大，但会降低脚手架作为可提交版本的完整度。

## 2. 目标

- 清理当前运行时仍可见的旧术语，统一为“选项组 / 可选项 / 选项配置”。
- 去掉已确认无用的字段或冗余代码，确保后端构建无当前已知 warning。
- 补齐本次收尾清理的文档和任务跟踪，形成可提交版本。

## 3. 非目标

- 不回头大规模重写历史设计文档中的旧术语。
- 不继续扩展功能范围，不新增页面或接口。
- 不整理与本次功能无关的其他脏工作区文件。

## 4. 方案对比

### 方案 A：只提交当前功能，不再清理

- 优点：最快。
- 缺点：运行时文案和构建 warning 会留到下一个版本，交付质量偏粗糙。

### 方案 B：做最小收尾清理后提交

- 优点：改动范围小，风险低，能明显提升版本完整度。
- 缺点：需要再做一轮验证和精确暂存。

### 方案 C：顺手全面整理全部历史文档与脏文件

- 优点：表面上更“干净”。
- 缺点：范围不可控，容易误碰无关内容，不适合当前收尾阶段。

## 5. 结论

采用方案 B。

本次只处理三类问题：

1. 当前页面、接口返回、README/TODO 中仍会误导用户的旧术语。
2. 已确认未使用、且不影响现有能力的字段或冗余代码。
3. 与本次版本直接相关的文档与验证闭环。

## 6. 风险与控制

- 删除未使用字段前，需确认前后端没有真实消费逻辑。
- 文案统一只改当前运行时与对外使用说明，不改历史方案沉淀，避免扩大 diff。
- 提交时只暂存本次相关文件，避免把用户其他未完成改动带入版本。

## 7. 验证

- `cargo check`
- `cargo test service::dict::tests -- --nocapture`
- `cargo test service::user_admin::tests -- --nocapture`
- `npm run build`
- `git diff --cached --stat`
