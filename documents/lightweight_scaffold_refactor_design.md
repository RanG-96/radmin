# 轻量脚手架可扩展性重构设计

## 1. 背景

当前脚手架已经具备中小型后台常见的基础模块：

- 登录 / 注册 / 用户管理
- 系统设置
- 文件管理
- 选项配置
- 消息通知
- 操作日志

整体技术选型和后端主分层是合理的，但随着页面数量增加，几个问题已经开始显现：

- 前端虽然有原子组件，但缺少脚手架级 CRUD 组合组件，`Users`、`DictTypes`、`DictItems` 等页面开始出现明显重复。
- 表单页已经覆盖新增、编辑、提交、更新，但字段排版、错误映射、提交态和控件补齐方式不统一。
- 前端接口与类型集中堆在一个 `api.ts` 中，短期方便，长期会降低可读性。
- 后端路由仍全部挂在 `main.rs` 中，当前还能接受，但再扩展会逐步变成长清单。

本项目定位是“面向中小型 Web 快速开发的轻量后台脚手架”，因此本轮目标不是做重型平台化能力，而是将已经稳定出现的重复模式沉淀成可复用骨架。

## 2. 目标

- 保持项目轻量，继续适合中小型后台快速起步。
- 收敛前端后台页面的重复骨架，降低新增 CRUD 页面的复制成本。
- 统一常见表单体验，让新增、编辑、保存类页面写法更一致。
- 将前端 API 和后端路由按领域拆分，提高可读性与可维护性。
- 尽量避免引入新的复杂框架或配置驱动体系。

## 3. 非目标

- 不做配置驱动整页 CRUD 平台。
- 不做低代码表单引擎。
- 不做复杂权限系统扩展。
- 不引入前端多层 repository / service / adapter 架构。
- 不替换现有 React、TanStack Query、Axum、SQLx 等核心技术栈。

## 4. 方案选型

### 4.1 方案 A：维持现状，仅在新增页面时复制现有模式

- 保留当前原子组件层。
- 继续在页面中手写搜索、表格、弹窗、分页和 mutation。

问题：

- 当前已经出现重复，后续只会继续扩散。
- 项目短期看似简单，长期会越来越依赖“复制再改字段”的开发方式。

### 4.2 方案 B：做轻量组合组件和最小结构拆分

- 前端补脚手架级组合组件，但不接管业务逻辑。
- 表单仍用 `useState`，补齐字段容器和错误映射。
- API 与类型按领域拆分，但仍保留薄封装。
- 后端只拆路由注册，不改现有 handler / service 主分层。

优点：

- 改动范围可控。
- 能直接降低重复代码。
- 符合“轻量脚手架”的定位。

### 4.3 方案 C：直接升级为配置驱动中后台框架

- 用列配置、字段配置、权限配置驱动整页 CRUD。
- 表单与表格都做高度抽象。

问题：

- 超出当前项目目标。
- 抽象成本和维护成本都明显上升。
- 不适合中小型项目的快速落地场景。

### 4.4 最终选择

采用方案 B。

## 5. 前端 CRUD 组合组件设计

本轮只收敛页面骨架，不把请求、状态和业务判断塞进组合组件。

### 5.1 PageHeader

用途：

- 统一页面标题、副标题和右侧主操作按钮布局。

解决的问题：

- 目前多个列表页都在重复写页头区域。

设计原则：

- 只负责结构和样式。
- 页面继续自己决定按钮点击行为。

### 5.2 FilterBar

用途：

- 统一搜索框、筛选项和次级操作区域布局。

设计原则：

- 不管理筛选状态。
- 只做容器和排版，让页面传入任意筛选控件。

### 5.3 DataTable

用途：

- 在现有 `Table` 原子组件之上，统一处理以下区域：
  - 头部筛选区
  - 加载态
  - 空态
  - 表格主体
  - 分页区

解决的问题：

- 当前每个列表页都在重复写相同的容器、边框、空态、加载态和分页结构。

边界：

- 不做列配置驱动。
- 不接管行渲染。
- 页面仍然自己写 `TableHeader` 和 `TableBody`。

### 5.4 RowActions

用途：

- 统一行内按钮区布局、间距和折行方式。

设计原则：

- 只负责排列，不封装业务动作。
- 优先支持显式按钮，不默认做三点菜单。

### 5.5 FormDialog

用途：

- 统一“弹窗标题 + 表单主体 + 关闭行为”的外层结构。

设计原则：

- 页面传入表单内容。
- 不引入复杂的插槽协议。
- 首轮只覆盖当前新增 / 编辑类弹窗场景。

### 5.6 StatusBadge

用途：

- 统一状态类展示：
  - 启用 / 禁用
  - 管理员 / 普通用户
  - POST / PUT / DELETE / PATCH 等方法状态

解决的问题：

- 当前各页面自己拼颜色 class，后续容易分散。

## 6. 统一表单方案设计

本轮不引入重型表单引擎，先收敛常见模式。

### 6.1 字段层

新增或补齐：

- `Textarea`
- `FormField` 容器，统一：
  - `label`
  - `hint`
  - `error`
  - `required`

保留现有：

- `Input`
- `FormSelect`
- `FormCheckbox`

收敛目标：

- 字段视觉和错误提示风格统一。
- 页面不再手写大量重复 `label + 输入框 + hint + error` 结构。

### 6.2 表单行为层

保持：

- 页面继续使用 `useState` 管理简单表单。

新增：

- 统一接口错误转表单文案的方法，例如：
  - `getApiErrorMessage` 继续复用
  - 或补轻量 `useFormError`

统一行为：

- 提交中按钮禁用
- 统一 `创建中 / 保存中 / 更新中` 文案
- 统一表单级错误展示位置

### 6.3 首轮覆盖页面

- `frontend/src/pages/Users.tsx`
- `frontend/src/pages/DictTypes.tsx`
- `frontend/src/pages/DictItems.tsx`
- `frontend/src/pages/Settings.tsx`

## 7. 前端 API 分模块设计

当前 `frontend/src/lib/api.ts` 同时承担：

- axios 实例
- 拦截器
- 所有领域类型
- 所有领域请求方法

本轮拆分为：

- `frontend/src/lib/http.ts`
  - 只放 axios 实例与拦截器
- `frontend/src/lib/types/`
  - 按领域拆类型，例如 `auth.ts`、`user.ts`、`dict.ts`
- `frontend/src/lib/api/`
  - 按领域拆请求，例如 `auth.ts`、`users.ts`、`dict.ts`、`files.ts`

原则：

- 不做自动生成。
- 不再套一层复杂仓储模式。
- 只做“按领域的薄文件拆分”。

收益：

- 页面 import 更清晰。
- 类型边界更清楚。
- 后续加模块不会继续堆进同一个文件。

## 8. 后端路由拆分设计

当前所有路由都直接注册在 `backend/src/main.rs`。

本轮只做轻量拆分：

- `backend/src/router/mod.rs`
- `backend/src/router/auth.rs`
- `backend/src/router/user.rs`
- `backend/src/router/admin.rs`
- `backend/src/router/dict.rs`
- `backend/src/router/file.rs`
- `backend/src/router/notification.rs`

职责：

- 每个 router 文件只负责组装本领域的 `Router`
- 不写业务逻辑
- 不改 handler / service 内部职责

`main.rs` 保留：

- 配置初始化
- 数据库初始化
- 中间件挂载
- 合并子路由

这样能让入口文件保持简洁，同时不引入多余复杂度。

## 9. 实施顺序

按以下顺序依次推进：

1. 前端 CRUD 组合组件层
2. 统一表单方案
3. 前端 API 分模块
4. 后端路由按领域拆分

原因：

- 先收骨架重复，再收表单重复，能最快体现脚手架价值。
- API 和路由拆分放在后面，更适合作为结构整理阶段。

## 10. 风险与控制

### 10.1 组件抽象过度

风险：

- 如果组合组件接管太多业务逻辑，会把项目做重。

控制：

- 每个组合组件只做结构层和展示层能力。

### 10.2 重构范围过大

风险：

- 一次性重写太多页面，容易引入回归。

控制：

- 首轮只覆盖用户、选项组、选项项、设置四类典型页面。

### 10.3 API 拆分引发 import 回归

风险：

- 路径调整后可能出现类型或请求引用遗漏。

控制：

- 每拆一个领域，立即做 lint / build 校验。

### 10.4 路由拆分影响现有行为

风险：

- 如果在路由拆分时顺手改动路径或中间件挂载，容易产生行为变化。

控制：

- 本轮只迁移注册代码，不改路径和中间件策略。

## 11. 验证策略

### 11.1 前端

- `yarn lint`
- 关键页面手动回归：
  - 用户管理
  - 选项组
  - 可选项
  - 设置页

重点确认：

- 页头、筛选区、表格区、弹窗区结构是否统一
- 新旧页面交互是否一致
- 表单错误提示和提交态是否正常

### 11.2 后端

- `cargo check`

重点确认：

- 路由拆分后路径不变
- 中间件仍正常挂载
- 编译通过且不引入新的领域耦合

## 12. 预期结果

完成本轮后，项目仍保持轻量，但会更接近“可持续二开的后台脚手架”：

- 新增一个常见 CRUD 页面的复制成本明显下降
- 页面组织方式更稳定
- 表单交互更统一
- 前后端结构更适合继续扩展
