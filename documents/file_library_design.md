# 轻量附件库增强设计

## 1. 背景

当前文件管理模块已经具备最小闭环：

- 上传
- 列表
- 下载
- 删除

但它更接近“附件上传示例页”，还不是适合二次开发的轻量文件管理模块。现有缺口主要集中在：

- 页面缺少搜索和筛选，文件一多就难以定位
- 上传区只有最基本的失败提示，缺少成功反馈和前置校验
- 列表没有上传人信息，也缺少复制链接和预览入口
- 上传大小和文件类型没有统一约束，后端也没有兜底校验

本项目定位仍然是中小型 Web 快速开发脚手架，因此本轮目标是把文件模块补成“可用的轻量附件库”，而不是做成完整素材中心。

## 2. 目标

- 保持当前文件模块轻量，不引入目录、分类等重型能力
- 补齐后台常见附件库最小体验：搜索、筛选、预览、复制链接、下载、删除
- 为上传增加前后端一致的类型和大小约束
- 让文件数据返回结构更适合页面展示和后续业务复用

## 3. 非目标

- 不做目录树
- 不做文件分类体系
- 不做重命名、移动、批量删除
- 不做复杂素材中心或图片墙
- 不做单独的文件权限系统

## 4. 方案选型

### 4.1 方案 A：仅补前端展示

- 前端增加搜索框、复制链接和预览
- 后端接口维持不变

问题：

- 搜索和筛选只能在当前页内做，真实数据量稍大就不成立
- 类型和大小限制只靠前端不可靠

### 4.2 方案 B：基于现有结构做轻量增强

- 继续复用当前 `files` 表和四个基础接口
- 为列表接口补搜索和类型筛选
- 为上传接口补大小和类型校验
- 为前端页面补常用附件库交互

优点：

- 改动范围小
- 不增加新的数据模型
- 能直接提升实际可用性
- 符合当前脚手架定位

### 4.3 方案 C：直接扩成素材管理

- 增加分类、目录、标签、批量操作

问题：

- 超出本轮需求
- 会明显拉高实现和维护复杂度

### 4.4 最终选择

采用方案 B。

## 5. 页面设计

页面继续基于当前 `frontend/src/pages/Files.tsx` 增量改造，不新增新的页面层抽象。

### 5.1 页面头部

- 标题仍为“文件管理”
- 副标题明确这是“轻量附件库”，用于上传、搜索和复用业务附件

### 5.2 筛选区

增加两个轻量筛选项：

- 文件名关键字搜索
- 文件类型筛选

筛选策略：

- 关键字按 `original_name` 做模糊匹配
- 文件类型先按大类分组：全部、图片、文档、音视频、其他

理由：

- 比直接暴露 MIME 字符串更易用
- 不需要新增额外字典配置

### 5.3 表格列

列表补齐以下信息：

- 文件名
- 文件类型
- 大小
- 上传人
- 上传时间
- 文件链接
- 操作

其中：

- 文件链接列提供“复制链接”快捷操作
- 操作列提供“预览 / 下载 / 删除”

### 5.4 空状态

区分两种空态：

- 无任何文件：提示先上传文件
- 有筛选条件但无结果：提示调整搜索条件

## 6. 上传设计

### 6.1 前端约束

`FileUpload` 增加：

- `accept`
- `maxSize`
- `helperText`
- 成功 / 失败提示

上传前先做：

- 文件类型白名单校验
- 文件大小上限校验

本轮默认支持：

- 图片：`image/*`
- 文档：`application/pdf`
- Office 常见格式
- 文本与压缩包

默认大小上限：

- 10MB

### 6.2 后端兜底

后端上传接口同步校验：

- 文件不能为空
- 文件大小不能超过 10MB
- MIME 类型必须在允许列表内

失败时返回明确中文错误，前端直接展示。

## 7. 后端接口设计

### 7.1 列表接口

保持现有路径：

- `GET /api/files`

新增查询参数：

- `page`
- `per_page`
- `keyword`
- `mime_type`

其中 `mime_type` 不是精确 MIME，而是页面传入的轻量分组值：

- `image`
- `document`
- `media`
- `other`

后端负责将分组值映射成 SQL 过滤条件。

### 7.2 上传接口

保持现有路径：

- `POST /api/files/upload`

新增行为：

- 在写盘前校验大小和类型
- 返回结构补上传人用户名，便于前端刷新列表后直接展示

### 7.3 下载与删除接口

路径不变：

- `GET /api/files/{id}/download`
- `DELETE /api/files/{id}`

本轮不修改删除和下载主行为。

## 8. 数据返回设计

当前 `FileResponse` 只有 `uploader_id`，不利于后台展示。调整为同时返回：

- `uploader_id`
- `uploader_name`

这样：

- 页面无需二次查用户表
- 后续业务表单中的附件选择器也能直接复用

## 9. 错误处理

前端统一处理以下场景：

- 上传文件类型不支持
- 上传文件超过大小限制
- 下载失败
- 删除失败
- 复制链接失败

设计原则：

- 使用现有轻量错误提示方式
- 错误信息尽量直接来自后端
- 不引入全局消息系统重构

## 10. 验证策略

后端验证：

- `cargo check`
- 已有本地运行实例下的接口冒烟：
  - 上传
  - 列表筛选
  - 下载
  - 删除

前端验证：

- `yarn lint`
- `yarn build`

人工回归重点：

- 搜索和类型筛选是否联动正确
- 上传限制和错误提示是否符合预期
- 复制链接和预览是否可用
- 删除后分页和列表刷新是否正常
