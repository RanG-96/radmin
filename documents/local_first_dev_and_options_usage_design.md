# 本地优先开发与选项配置使用说明设计

## 1. 背景

当前项目虽然已经具备 Docker Compose 方案，但用户希望现阶段默认开发模式改为：

- PostgreSQL 使用本地安装实例
- 后端单独启动
- 前端单独启动
- `docker-compose.yml` 保留，但不再作为默认开发路径

同时，选项配置页面虽然已经做了管理员易用性重构，但由于不新增业务示例页，仍需要通过文档方式把“管理员如何配置、开发者如何消费”讲清楚。

## 2. 目标

- 将项目默认开发路径改成“本地优先”，降低对 Docker 的依赖。
- 保留 Docker Compose 作为可选部署或备用运行方式。
- 通过 README 和项目文档补齐选项配置的使用说明，避免没有示例页时功能价值不清楚。

## 3. 非目标

- 不删除 `docker-compose.yml`
- 不删除 Dockerfile
- 不新增新的业务演示页面
- 不改动后端运行机制，只改默认使用方式和文档描述

## 4. 方案

### 4.1 开发模式调整

- `Makefile` 的 `dev` 不再自动 `docker compose up -d postgres`
- `dev` 只负责并行启动后端与前端，假设本地 PostgreSQL 已可用
- Docker 相关命令继续保留为可选项

### 4.2 README 信息重排

- 将“本地开发”提升为默认推荐路径
- 将“Docker 启动”降为可选方案
- 补充本地 PostgreSQL 的准备说明：
  - 数据库名
  - 默认连接串
  - migration 命令
- 明确前后端分开启动方式

### 4.3 选项配置使用说明

由于不做示例页，本次通过文档闭环：

- 说明“选项配置”是做什么的
- 说明管理员如何使用：
  - 创建选项组
  - 添加可选项
  - 启用/禁用
- 说明开发者如何消费：
  - 接口 `/api/dict/{type_code}`
  - 前端组件 `DictSelect`
  - 保存 `value`，展示 `label`

## 5. 设计取舍

### 5.1 为什么不删 Docker

- 保留容器化部署能力，方便后续联调、验收或部署。
- 当前只是将其从“默认开发路径”降级为“可选方案”。

### 5.2 为什么不用示例页来补闭环

- 用户已明确不需要示例页。
- 对脚手架而言，文档说明比引入一个半业务化示例页面更轻量。

## 6. 风险

- 如果开发者本地没有 PostgreSQL，`make dev` 将无法直接成功。
- 因此 README 必须把本地数据库准备步骤写清楚。

## 7. 验证

- `make dev-backend`、`make dev-frontend` 命令描述与 README 保持一致
- `make dev` 不再依赖 Docker PostgreSQL
- README 中本地开发为主，Docker 为辅
- 选项配置的管理员使用方式与开发者消费方式均有清晰文档
