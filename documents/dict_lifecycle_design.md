# 数据字典生命周期设计

## 1. 背景

当前数据字典功能已经具备基础 CRUD，但还不满足“稳定可靠、可复用脚手架能力”的目标，主要问题如下：

- `dict_items` 在同一 `dict_type_id` 下缺少 `value` 唯一性约束，消费端又直接使用 `value` 作为选择值，存在业务歧义。
- 字典类型删除为级联硬删，风险过高，不适合作为数据字典主操作。
- 前后端都有 `status` 字段，但页面没有启用/禁用入口，生命周期设计不完整。
- 创建、编辑、删除失败时缺少明确的用户可见反馈。

## 2. 目标

- 让数据字典成为可长期复用的脚手架能力，而不是一次性管理页。
- 明确类型与字典项的生命周期规则，降低误删风险。
- 强化后端约束，保证前端交互建立在可靠的数据规则之上。
- 保持现有架构和 Radix UI 组件体系，不引入额外复杂度。

## 3. 非目标

- 不引入拖拽排序。
- 不引入软删除字段、归档恢复、审计追溯等更重的生命周期系统。
- 不改造字典对业务侧的消费方式，仍然按 `type_code` 查询启用项。

## 4. 方案选型

### 4.1 备选方案

#### 方案 A：软生命周期 + 受限删除

- 类型和字典项都以 `status` 作为主生命周期控制。
- 类型和字典项都支持启用/禁用。
- 类型仅在没有任何字典项时允许删除。
- 字典项仍允许删除，但启用/禁用为推荐主路径。

#### 方案 B：类型只允许启停，字典项保留硬删

- 类型完全不可删，只能启用/禁用。
- 字典项允许删除，也允许启停。

#### 方案 C：全面软删除/归档

- 类型与字典项都不做物理删除，只做逻辑归档。

### 4.2 最终选择

采用方案 A。

原因：

- 与现有数据模型最一致，改动小。
- 能直接修复当前 review 中的核心问题。
- 既保留管理灵活性，也避免把高风险硬删设为主路径。

## 5. 生命周期规则

### 5.1 字典类型

- 创建时默认 `status = true`。
- 编辑时允许修改名称、备注、状态。
- `type_code` 仍允许作为稳定识别码存在，编辑态默认只读，避免影响业务消费端。
- 删除仅允许在该类型下没有任何字典项时进行。
- 若仍存在字典项，前端禁用删除按钮，后端也必须拒绝删除请求。

### 5.2 字典项

- 创建时默认 `status = true`。
- 编辑时允许修改标签、值、排序、备注、状态。
- 启用项可被业务端消费；禁用项不出现在 `/api/dict/{type_code}` 的返回中。
- 字典项允许删除，但启停是更推荐的生命周期操作。

## 6. 数据与接口设计

### 6.1 数据约束

- 在 `dict_items` 上增加 `(dict_type_id, value)` 唯一约束。
- 保留 `dict_type_id` 普通索引，用于列表查询。

### 6.2 后端校验

- `create_item` / `update_item` 增加同类型下 `value` 重复校验，返回明确校验错误。
- `delete_type` 执行前检查是否存在关联字典项；若存在则拒绝删除。
- `update_type` / `update_item` 使用现有 `status` 字段完成启停。

### 6.3 后端返回

- 保持现有 CRUD 路径不变，避免前端路由层大改。
- 新增按类型 ID 获取字典类型详情的轻量接口，用于字典项页头展示所属类型上下文。
- 统一返回可直接展示给用户的错误消息，例如：
  - `类型编码已存在`
  - `该字典类型下的字典值已存在`
  - `该字典类型下仍有字典项，不能删除`

## 7. 前端页面设计

### 7.1 字典类型页

- 保留搜索和分页。
- 操作区调整为：`字典项`、`编辑`、`启用/禁用`、`删除`。
- 删除按钮仅在没有字典项时可用，否则禁用并提示原因。
- 新增/编辑弹窗补充状态控件。
- 删除确认文案只针对空类型，不再提示级联删除。

### 7.2 字典项页

- 页头展示所属字典类型名称和 `type_code`，增强上下文。
- 操作区调整为：`编辑`、`启用/禁用`、`删除`。
- 新增/编辑弹窗补充状态控件。
- 保留排序字段为数字输入，优先保证稳定性。

### 7.3 错误与反馈

- 每个 mutation 都提供提交中状态。
- 每个弹窗内提供失败提示区域，显示后端返回错误。
- 成功时关闭弹窗并刷新列表。

## 8. 组件与实现约束

- 继续沿用现有 Radix UI primitive 封装，不引入新的 UI 库。
- 优先复用：
  - `Dialog`
  - `AlertDialog`
  - `Select`
  - `Checkbox`
- 如果状态控件需要更明确的开关语义，可以新增一个轻量 `Switch` 包装，但仍基于 Radix primitive。

## 9. 风险与回归点

- 新增唯一约束前，需要考虑已有重复数据导致 migration 失败的风险。
- 禁用类型或项后，业务侧下拉选项会立即减少，需要确认这是预期行为。
- 删除规则收紧后，前端和后端必须同步，否则会出现 UI 与接口行为不一致。

## 10. 验证策略

- 后端：
  - 校验重复字典值时返回正确错误。
  - 校验非空类型删除被拒绝。
  - 校验禁用类型/项后 `/api/dict/{type_code}` 只返回启用数据。
- 前端：
  - 类型页和字典项页都能启用/禁用。
  - 删除按钮门禁正确。
  - 失败提示能展示后端错误信息。
  - 字典项页能显示所属类型上下文。
