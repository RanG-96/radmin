# 管理员易用性重构设计

## 1. 背景

当前后台脚手架已经具备用户管理、数据字典、通知、设置等基础能力，但在普通后台管理员视角下，仍存在明显的认知与操作阻力：

- “数据字典 / 字典类型 / 字典项”偏技术术语，页面用途不自解释。
- 数据字典只有管理端，没有业务消费示例，用户看不到“配置完会用在哪里”。
- 用户管理页将操作藏在三点菜单里，缺少显式操作列和高频动作前置。
- 前端多个管理页的错误反馈模式不一致。
- 后端整体技术栈合理，但部分领域逻辑直接堆在 handler 中，模块边界还不够清晰。

## 2. 目标

- 让普通后台管理员无需理解 `type_code`、`字典` 等开发术语，也能理解和使用配置功能。
- 让用户管理页具备清晰、直接、可预期的行内操作。
- 在不推翻现有技术栈的前提下，收敛后端模块边界，让结构更清楚、更易维护。
- 保持页面简洁，不引入过度复杂的交互模型。

## 3. 非目标

- 不重做成复杂的单页主从工作台。
- 不替换核心后端框架或 ORM。
- 不为所有领域一次性做大规模架构迁移。
- 不在本次实现中补完整业务消费页，但要让管理页足够自解释，并为后续消费示例留出清晰接口。

## 4. 方案选型

### 4.1 备选方案

#### 方案 A：仅改文案与按钮位置

- 保留“数据字典 / 字典项”命名。
- 只补说明文案与显式操作列。

问题：

- 仍然保留术语门槛。
- 对普通管理员不够自然。

#### 方案 B：改成“选项配置 / 选项组 / 可选项”模型

- 将技术术语替换成面向管理员的语言。
- 保留两层结构，但把信息架构、页头说明、空状态、行内操作都改得更直观。
- 用户管理页同步改成显式操作列。
- 后端保留现有技术栈，仅做模块收敛与输入校验增强。

优点：

- 改动范围可控。
- 能明显提升可理解性与可用性。
- 适合脚手架长期复用。

#### 方案 C：选项配置做成单页主从编辑台

- 左侧选项组，右侧即时编辑组选项。
- 后端同时做更深层模块改造。

问题：

- 改动大。
- 超出本轮“简洁可用”的最小闭环。

### 4.2 最终选择

采用方案 B。

## 5. 前端信息架构

### 5.1 选项配置

- 侧边导航与仪表盘入口统一改名为 `选项配置`。
- 原“数据字典”列表页在交互上改造成“选项组列表”：
  - 选项组
  - 用途说明
  - 可选项数
  - 状态
  - 操作
- `type_code` 作为技术标识保留，但降级成次级信息。
- 页头增加用途说明，明确它是“管理表单中的下拉选项、状态选项和枚举配置”。

### 5.2 可选项管理

- 原“字典项页”改造成“管理可选项”页面。
- 页头显示面包屑式上下文：`选项配置 / 组选项名称`。
- 辅助说明显示当前组选项的用途说明和系统编码。
- 列表列保持简洁：
  - 显示名称
  - 实际值
  - 排序
  - 状态
  - 操作

### 5.3 用户管理

- 增加明确的 `操作` 列标题。
- 高概率动作前置成可见按钮：
  - 编辑
  - 启用/禁用
  - 删除
- 去掉“必须点三点菜单才能操作”的模式。
- 顶部搜索与新增保留。

## 6. 前端交互设计

### 6.1 选项组表单

- `名称` 改为 `选项组名称`。
- `备注` 改为 `用途说明`。
- `类型编码` 改为 `系统编码`，并提示“供系统内部使用”。
- 编辑态仍禁止修改系统编码，避免破坏业务消费侧。

### 6.2 可选项表单

- `标签` 改为 `显示名称`。
- `值` 改为 `实际值`。
- `备注` 改为 `补充说明`。

### 6.3 空状态与说明

- 选项配置页空状态明确提示“先创建一个选项组，再添加可选项”。
- 可选项页空状态明确提示“当前组选项还没有可选项，先新增一个”。
- 页头说明避免出现数据库或开发术语。

### 6.4 错误反馈

- 选项配置页、可选项页、用户管理页统一采用：
  - 页级错误条
  - 弹窗内表单错误提示
  - 提交中按钮禁用

## 7. 后端重构策略

## 7.1 保持现有核心库

不替换以下库：

- `axum`
- `sqlx`
- `jsonwebtoken`
- `bcrypt`

原因：

- 它们已经是成熟且适合当前体量的方案。
- 当前问题更偏组织结构，而不是库能力不足。

### 7.2 适度引入 `validator`

- 为用户管理及后续适合的 DTO 增加声明式输入校验。
- 用于统一邮箱格式、长度、必填等规则。
- 减少散落在 handler 中的手写校验。

### 7.3 模块边界收敛

- 将用户管理中的 SQL 与业务校验从 `handler/user_admin.rs` 下沉到独立 service 模块。
- 让 handler 更像“接收请求 + 调用 service + 返回响应”。
- 保持 `handler -> service -> database` 主链不变。

### 7.4 路由组织优化

- 不在本次引入复杂 router 框架。
- 但可将路由注册按领域提取为独立函数，减少 `main.rs` 的堆叠感。

## 8. 风险

- 前端文案变更会影响已有用户记忆，需要保持词汇统一。
- 选项配置与用户管理页同时改动，需注意交互风格一致。
- 引入 `validator` 后，错误文案需要保持对中文用户友好。

## 9. 验证策略

- 前端：
  - 选项配置入口与页面命名统一
  - 选项组与可选项页面信息层级更清晰
  - 用户管理显式操作列可正常工作
  - 错误提示与提交态正常
- 后端：
  - 用户管理 service 下沉后接口行为不变
  - `validator` 校验生效
  - 构建通过，现有 API 不回归
